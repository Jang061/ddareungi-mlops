# 문제 정의 (Week 1 Day 1)

## 1. 예측 타깃 (Target)

| 항목 | 내용 |
|------|------|
| **타깃 변수** | `ride_count` — 특정 대여소에서 특정 시간대에 발생한 대여 건수 |
| **그레인 (grain)** | (대여소 ID, 날짜, 시간대) — 1행 = 1대여소 × 1시간 |
| **예측 지평선** | T+1h (1시간 후 수요 예측) |
| **예측 단위** | 시간(hour) — 시간대별 집계 |

## 2. 평가 지표

| 지표 | 선택 이유 |
|------|-----------|
| **RMSE** (Root Mean Squared Error) | 주요 지표. 큰 오차에 페널티, 원본 단위(건)로 해석 가능 |
| **MAE** (Mean Absolute Error) | 보조 지표. 이상치에 덜 민감, 실운용 감각 파악용 |
| **MAPE** | 대여가 0인 시간대에서 정의 불가 → 사용 안 함 |

## 3. Naive 베이스라인

비교 기준이 없으면 모델 개선을 논할 수 없다. 두 가지 naive baseline을 정의한다.

### Baseline-1: 직전 시간 그대로 (Lag-1)
```
ŷ(station, t) = y(station, t-1)
```
- 구현 난이도: 매우 낮음
- 의미: "지금과 비슷하겠지" 가정

### Baseline-2: 같은 요일·같은 시간 지난주 평균 (Seasonal Naive)
```
ŷ(station, t) = mean( y(station, 같은 요일·시간대) ) — 최근 4주
```
- 구현 난이도: 낮음
- 의미: 요일·시간 패턴만 반영

> **목표**: LightGBM 모델이 두 baseline 모두를 RMSE 기준 명확히 하회해야 한다.

## 4. 사용 피처 초안

### 시간 피처 (데이터 누수 없음)
- `hour`, `day_of_week`, `is_weekend`, `month`, `is_holiday`

### 날씨 피처 (예측 시점의 예보값 사용 — 과거 실측 아님)
- `temp` (기온), `rainfall` (강수량), `wind_speed` (풍속), `humidity`
- 주의: 서빙 시에도 예보 API 호출로 동일하게 구성 (스큐 방지)

### 대여소 피처
- `station_id` (카테고리 인코딩 또는 embedding)
- `district` (자치구)

### Lag / Rolling 피처 (누수 주의)
- `lag_1h`, `lag_24h`, `lag_168h` (1주 전 동시간)
- `rolling_mean_24h` (직전 24시간 평균)
- ⚠️ 예측 시점 기준 과거값만 사용. 미래 정보 절대 금지.

## 5. 데이터 범위

| 구분 | 기간 | 비고 |
|------|------|------|
| **학습** | ~ 검증 시작 2주 전 | 시계열 순서 고정 |
| **검증** | 마지막 4주 중 앞 2주 | 하이퍼파라미터 튜닝용 |
| **테스트** | 마지막 2주 | 최종 성능 보고용, 학습 중 절대 미사용 |

> 랜덤 분할 금지. 미래 데이터가 학습에 유입되면 누수.

## 6. 성공 기준

1. LightGBM RMSE < Baseline-2(Seasonal Naive) RMSE
2. MLflow에 모든 실험 run이 기록되어 있고 재현 가능
3. `python train.py` 한 줄로 학습 완료
